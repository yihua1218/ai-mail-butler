# WebLLM Gmail 瀏覽器 Extension 可行性評估

## 執行摘要

本文件評估一種產品設計：在 Chrome 或 Edge 瀏覽器 Extension 內透過 WebLLM 執行本地端大型語言模型，協助使用者讀取 Gmail 郵件、起草回覆，並在特定條件下代為送出，而不將信件內容轉送到第三方推論伺服器。

此構想在技術上可行，但只能在受控的發布模型下落地。以本地優先的草稿助手來說，可行度高；若要做到全自動回信，則只有在授權流程、瀏覽器生命週期、模型效能與失效處理都補強後，才具有中等可行性。

## 目標成果

- 讓使用者的信件內容、提示上下文、規則與語氣設定都留在使用者裝置上的瀏覽器內。
- 使用 WebLLM 在本地端進行推論，不將郵件內容送到外部 AI 供應商。
- 提供一個以 Gmail 為主的瀏覽器 Extension，可協助收信閱讀、回信草稿生成，並在條件允許時代為回信。
- 允許使用者在本地端定義規則、語氣、審核門檻與自動化範圍。

## 範圍與假設

### 納入範圍

- V1 僅支援 Chrome 與 Microsoft Edge。
- 以 Gmail 網頁版為主要使用介面。
- WebLLM 以瀏覽器內的 WebGPU 加速執行。
- 所有設定、規則、prompt 模板與本地稽核資料皆儲存在瀏覽器本地。
- 以使用者授權的瀏覽器操作方式完成收件檢視、回信草稿與送信。

### 不納入範圍

- Firefox、Safari 與手機瀏覽器。
- 伺服器端推論。
- 提示、設定或信件歷史的多租戶雲端同步。
- V1 的企業管理員深度整合。

### 運作假設

1. 使用者已在安裝 Extension 的瀏覽器設定檔中登入 Gmail。
2. 使用者接受 Extension 需要讀取頁面內容與本地儲存設定的權限模型。
3. 使用者裝置與瀏覽器對 WebGPU 的支援足以執行至少一個 WebLLM 相容模型。
4. V1 預設為草稿優先模式，只有在使用者明確批准的規則子集合中才允許自動送出。

## 想達成的狀態

### 產品目標

1. 以低摩擦方式讀取與回信相關的 Gmail 對話內容。
2. 依據使用者偏好的語氣與格式，在本地端產生回覆。
3. 將設定與由郵件衍生出的工作資料維持在瀏覽器設定檔本地。
4. 提供足夠穩定性，讓使用者可以日常使用，而非一直手動救援。

### 隱私目標

1. 不將郵件本文送到遠端模型推論 API。
2. 不需要集中式應用伺服器來分析郵件或生成回覆。
3. 預設所有日誌都保留在本地，並提供遮罩與保留期控制。
4. 使用者可以隨時檢視與刪除本地資料。

## 可行度摘要

| 面向                        | 評級 | 說明                                                                             |
| :-------------------------- | :--- | :------------------------------------------------------------------------------- |
| WebLLM 本地推論             | 高   | WebLLM 明確支援 Chrome Extension、dedicated worker 與 service worker。           |
| 本地設定與規則儲存          | 高   | `chrome.storage.local`、IndexedDB 與 Cache API 足以保存設定、prompt 與模型快取。 |
| Gmail 草稿輔助              | 高   | 以 content script 協助的流程，對於有人監督的草稿使用情境已足夠可靠。             |
| Gmail API 整合              | 中   | 技術上完整，但需要明確權杖流程與較正式的授權路徑。                               |
| 全自動送信                  | 中低 | 只有在安全閘門、生命週期復原與信任邊界設計完整時才可行。                         |
| 隱私承諾                    | 中高 | 對 AI 處理流程來說很強，但仍受 Gmail、Extension 權限與瀏覽器本地安全限制。       |
| Chrome Web Store 上架準備度 | 中   | 可以做到，但 Extension 行為必須聚焦且透明，才能降低審查風險。                    |

## 候選架構

### 方案 A：Gmail API 加上 `chrome.identity` 加上 WebLLM

#### 方案 A 流程

1. Extension 透過瀏覽器 identity flow 取得 Gmail 存取權杖。
2. 透過 Gmail API 列出 thread、取得信件內容、建立草稿或送出回覆。
3. 郵件內容在本地經由 WebLLM 摘要與生成回覆。
4. 本地規則決定要建立草稿還是自動送出。

#### 方案 A 優點

- 整合方式最穩定、結構化。
- 可清楚使用 thread、label、draft 與 message metadata。
- 更容易處理冪等、重試邏輯與 thread 關聯。
- 長期維護性優於 DOM 自動化。

#### 方案 A 缺點

- 需要完整授權流程與清楚的權限 UX。
- 初次 onboarding 摩擦較大。
- 因為會明確存取信箱資料，可能增加商店審查與使用者同意的敏感度。

#### 方案 A 評估

若產品目標是穩定且可維護，這是最好的工程架構。它應該是生產級行為的目標架構。

### 方案 B：Gmail 網頁 UI 自動化加上 Content Script 加上 WebLLM

#### 方案 B 流程

1. 在 `mail.google.com` 執行 content script。
2. 從目前打開的對話串或收件匣畫面讀取信件內容。
3. 側欄或浮動 compose 助手呼叫 WebLLM 在本地生成回覆。
4. Extension 將內容填入 Gmail compose 視窗，必要時再觸發原生送出動作。

#### 方案 B 優點

- 不需要額外的伺服器端整合。
- 原型開發速度快。
- 即使 API-based onboarding 延後，仍可作為使用者可見的瀏覽器助手。

#### 方案 B 缺點

- Gmail DOM 結構變動時容易失效。
- 不容易穩定支援背景掃描收件匣。
- 對 thread 狀態的自動化較難做到可預測。
- Gmail UI 調整後較容易出現靜默失敗。

#### 方案 B 評估

這是最快的 MVP 路徑，也適合有人監督的草稿助手；但它不是無人值守自動化的最佳基礎。

### 方案 C：Hybrid 設計

#### 方案 C 流程

1. 可用時以 API 處理穩定的信箱操作。
2. 以 content script 提供頁內 compose 輔助、inline 控制與可見的審閱 UX。
3. WebLLM 與規則引擎都維持在 Extension 本地執行。

#### 方案 C 優點

- 在體驗與可靠性之間取得最佳平衡。
- 頁內助手可維持直覺，信箱操作則更具結構性。
- 可以從監督式草稿逐步過渡到選擇性自動化。

#### 方案 C 缺點

- 工程複雜度最高。
- 必須仔細處理 API 狀態與頁面 UI 狀態的一致性。

#### 方案 C 評估

這是建議的目標架構。它允許產品以務實方式分階段成長：先做 UI 助手，再逐步補上 API 帶來的穩定性。

## 建議方向

採用方案 C 作為策略性架構，但交付順序應如下：

1. Phase 0：先驗證 WebLLM 在 Chrome 或 Edge Extension 內的效能與記憶體行為。
2. Phase 1：先交付有人監督的 Gmail 網頁草稿輔助，不提供無人值守送出。
3. Phase 2：加入 API-backed 草稿建立與可預測的 thread 處理。
4. Phase 3：只對高信心、明確收斂的規則類型開放選擇性 auto-send。

## V1 詳細產品規格

### 主要使用流程

1. 使用者安裝 Extension，並在 Chrome 或 Edge 中開啟 Gmail。
2. Extension 開啟側欄，提供模型設定、規則、語氣 preset 與安全設定。
3. 使用者選擇本地模型，等待第一次下載與 warm-up。
4. 當使用者開啟某個 thread 時，Extension 提供以下動作：摘要 thread、建議回覆、建議簡短回覆、根據規則建立草稿，以及將寄件者標記為可安全自動化或僅人工審核。
5. 生成的回覆會並列顯示說明、信心分數、命中的規則與送信模式。
6. 預設情況下，必須由使用者核准後才會插入或送出。

### 可選自動化模式

- `manual`：僅提供建議文字，由使用者自行貼上或插入。
- `draft_assist`：Extension 可寫入 Gmail 草稿，但絕不自動送出。
- `guarded_autosend`：只有在明確 allowlist 規則與信心門檻滿足時，才可自動送出。
- `disabled_for_sender`：對此寄件者、label 或網域永不自動化。

### 規則引擎需求

每條規則至少應支援以下欄位：

| 欄位                       | 用途                       |
| :------------------------- | :------------------------- |
| `id`                       | 穩定的本地識別碼           |
| `enabled`                  | 啟用或停用規則             |
| `priority`                 | 解決規則衝突               |
| `match.sender`             | 信箱或網域比對             |
| `match.labels`             | Gmail label 或推斷 label   |
| `match.subject_regex`      | 主旨規則                   |
| `match.contains`           | 關鍵字或片語比對           |
| `tone_profile_id`          | 對應語氣設定               |
| `instruction_template`     | 使用者自訂模型指示         |
| `allowed_actions`          | 僅草稿或可 auto-send       |
| `min_confidence_auto_send` | 自動送出的最低信心門檻     |
| `cooldown_minutes`         | 避免重複動作               |
| `requires_human_review`    | 對敏感 thread 強制人工審核 |

### 語氣設定需求

每個 tone profile 應支援：

- 顯示名稱
- System prompt 模板
- 偏好的問候語與結尾語
- 允許的語言清單
- 偏好的信件長度
- 正式程度
- 是否傾向主動提問
- 是否避免在未核准下做出承諾

### 安全控制

1. 所有新規則預設都為 `draft_assist`。
2. 對含附件、發票、法律關鍵字、人資議題、退款爭議或醫療內容的信件，預設阻擋 auto-send，除非使用者明確 allowlist。
3. 必須有本地 cooldown，避免同一 thread 重複自動回覆。
4. 顯示說明面板，列出命中的規則、寄件者、信心分數與模型為何產生此回覆。
5. 預設只記錄本地決策摘要，不保存完整明文 thread 內容，除非使用者明確選擇開啟。

## 技術架構

### 建議的 Runtime 佈局

- `extension service worker`：負責 alarms、規則排程、權杖刷新狀態與背景事件。
- `content script`：讀取 Gmail UI 狀態、插入操作元件、橋接 compose 動作。
- `side panel`：提供設定 UI、模型選擇、稽核檢視與復原操作。
- `WebLLM worker 或 service worker`：將模型載入與推論從 UI thread 移開。
- `IndexedDB` 加上 `chrome.storage.local`：保存規則、設定、語氣設定與本地稽核 metadata。
- `Cache API`：在使用 WebLLM cache backend 時保存模型 artifacts。

### 為何要這樣拆分

- Manifest V3 的 service worker 是事件驅動，閒置時可能被卸載。
- 高負載推論不應該放在 content script 裡。
- 模型載入必須能承受 UI 切換，且在 worker 被卸載時要能安全降級。

### 郵件處理流程

1. 觸發來源發生：開啟 thread、打開 compose、alarm tick 或使用者命令。
2. 將內容正規化為本地 prompt bundle。
3. 規則引擎判定模式、語氣設定與是否符合處理資格。
4. WebLLM 生成結構化結果，其中包含回覆文字、信心分數、敏感性旗標與建議動作。
5. 政策層決定為：忽略、顯示建議、建立草稿或送出。
6. 本地稽核層記錄 metadata 與結果。

## 本地資料模型

### 本地儲存分類

| 類別               | 儲存位置               | 說明                            |
| :----------------- | :--------------------- | :------------------------------ |
| Extension 設定     | `chrome.storage.local` | 小型設定、開關、onboarding 狀態 |
| 規則與語氣設定     | IndexedDB              | 較適合 schema 演進與查詢        |
| 模型 artifacts     | Cache API 或 IndexedDB | 由 WebLLM cache backend 管理    |
| 本地稽核 metadata  | IndexedDB              | 預設遮罩                        |
| 暫存 prompt bundle | 僅記憶體               | 請求完成後清除                  |

### 強烈建議

對以下敏感本地資料，使用 WebCrypto 加上使用者自訂 passphrase 做靜態加密：

- 含敏感業務邏輯的規則模板
- Allowlists 與寄件者分類
- 超過最小遙測範圍的本地稽核 metadata

## 隱私邊界評估

### 留在本地的內容

- Prompt 組裝
- 模型推論
- 使用者自訂規則與語氣設定
- 信心評估與自動化決策
- 啟用時的本地稽核紀錄

### 不會因此改變的事實

- Gmail 仍然是郵件提供者，因此郵件內容本來就存在 Google 自身的產品與基礎設施中。
- 如果模型權重是從遠端主機下載，瀏覽器仍會聯絡該主機抓取 artifacts，但不包含郵件內容。
- 具有信箱讀取權限的瀏覽器 Extension，依然是使用者本機上的高信任元件。

### 可成立的隱私主張

若預設關閉遙測，且 debug log 做好遮罩，則本產品可以合理主張：回信生成與自動化判斷邏輯都在瀏覽器本地執行，郵件內容不會再轉送給第三方 AI 推論服務。

## 主要風險與必要緩解措施

| 風險                               | 影響 | 機率 | 緩解措施                                                                     |
| :--------------------------------- | :--- | :--- | :--------------------------------------------------------------------------- |
| 使用者裝置的 WebGPU 不可用或不穩定 | 高   | 中   | 提供相容性檢查；若無支援 runtime，則退回監督式模式。                         |
| 模型冷啟動太慢                     | 高   | 高   | 先採用較小的 WebLLM 相容模型、按需預熱、積極快取並清楚顯示進度。             |
| Extension worker 卸載導致推論中斷  | 中   | 高   | 將推論移至 dedicated worker 或 service-worker flow，並實作可重試的狀態復原。 |
| Gmail UI 變更導致 DOM 自動化失效   | 高   | 中   | 限縮 UI 自動化範圍、加入 selector 健康檢查，並逐步把穩定操作移到 API。       |
| 在敏感 thread 上錯誤自動回覆       | 高   | 中   | 預設草稿優先、加入內容風險過濾，且 auto-send 僅允許 allowlist 規則。         |
| 共用電腦上的本地儲存洩漏           | 中   | 中   | 支援以 passphrase 為基礎的本地加密與一鍵清除。                               |
| 商店審查或權限申請被拒             | 高   | 中   | 權限最小化、清楚說明使用者利益，並在產品中提供透明揭露。                     |
| 模型品質不足以處理細膩回覆         | 中   | 高   | 初期只鎖定重複性高、低風險的郵件類別，並保留人工審核模式。                   |

## 需要克服的工程問題

### 1. 瀏覽器 Runtime 限制

Manifest V3 的背景執行不是常駐的。Extension 必須能承受 service worker 暫停、tab 關閉與模型重新載入成本。

### 2. 模型大小與延遲

WebLLM 可以本地執行，但實際體驗取決於模型大小、量化方式、GPU 品質、可用記憶體，以及第一次載入 artifact 的下載時間。

### 3. 結構化輸出可靠性

若要支援自動化，自由文字不夠。Extension 應要求模型輸出結構化 JSON，至少包含回覆文字、信心分數、敏感旗標與建議動作。

### 4. 送信安全邊界

送出信件是高影響動作。政策層必須獨立於模型，不能讓模型自行決定是否有送信權限。

### 5. Gmail 存取策略

純 DOM 自動化太脆弱；純 API 存取雖更穩定，但 onboarding 與權限複雜度較高。產品應採取分階段 hybrid 路線。

## 各里程碑的實務可行性

### Phase 0：技術驗證

目標：驗證 runtime 可行性。

交付項目：

1. 在 Chrome 或 Edge Extension 中載入一個 WebLLM 模型。
2. 對畫面上可見的 Gmail thread 做摘要示範。
3. 量測冷啟動、暖機後延遲、記憶體使用量與 worker 復原行為。

驗收條件：

1. 至少在一種常見 Chrome 與一種常見 Edge 測試環境中成功初始化模型。
2. 暖機後的中位數回覆延遲足以支撐草稿輔助。
3. 頁面刷新後 Extension 不會破壞本地狀態。

### Phase 1：監督式草稿助手

目標：交付安全的本地優先 MVP。

交付項目：

1. 用於規則、語氣設定與模型狀態的側欄 UI。
2. 以 content script 擷取 thread 並插入草稿。
3. 本地稽核 metadata 與一鍵刪除控制。

驗收條件：

1. 使用者可對指定 thread 生成草稿回覆。
2. 尚不存在 auto-send 路徑。
3. 所有設定皆保留在瀏覽器設定檔本地。

### Phase 2：穩定草稿流程

目標：提高真實使用情境下的可靠性。

交付項目：

1. 在可用時加入 API-backed 草稿建立。
2. 可預測的 thread 對應與 cooldown 強制執行。
3. 含安全分類的結構化 JSON 生成。

驗收條件：

1. 草稿建立可重現，且對 Gmail UI 變更更具韌性。
2. 規則引擎可正確擋下敏感 thread。
3. Worker 被卸載後的復原行為具可預測性。

### Phase 3：選擇性受控 Auto-Send

目標：在強控制條件下開放窄範圍自動化。

交付項目：

1. Allowlist 的寄件者與規則類型。
2. 信心門檻與安全否決層。
3. Auto-send 決策的本地稽核軌跡。

驗收條件：

1. Auto-send 只對明確核准的規則類型開放。
2. 敏感內容偵測能擋下不安全 thread。
3. 使用者可立即停用自動化並清除本地狀態。

## 最終評估

### 整體可行度

此構想可行。

### 依產品形態區分的可行度

- 本地 Gmail 草稿助手：高
- 本地規則驅動回覆生成：高
- 瀏覽器內的背景信箱分流：中
- 不經審核的可信全自動回信：中低

### 為何可行度不是更高

主要限制不是 WebLLM 能不能在本地跑。它可以。真正困難的是 Extension 生命週期控制、不同使用者裝置間的效能差異、Gmail 整合的耐久性，以及如何把安全閘門做得足夠強，讓無人值守送信仍可被信任。

## 建議

這個產品值得做，但不應從自動送信開始。

建議發布順序：

1. 先做純本地、有人監督的草稿助手。
2. 第一批只鎖定重複性高、低風險的郵件類型。
3. 先補齊穩定整合與結構化安全閘門，再開放 auto-send。
4. 將全自動送信定位為進階模式，要求明確本地 opt-in 與可逆控制。

## 驗證清單

1. 驗證 WebLLM 在 Manifest V3 Extension 內的啟動、復原與暖機延遲。
2. 驗證所有設定與規則資料皆保留在本地，且可隨時刪除。
3. 驗證預設情況下，沒有任何 debug 或 telemetry 路徑會將郵件本文外送。
4. 驗證敏感 thread 偵測可阻止被封鎖類別的 auto-send。
5. 驗證 Gmail UI 改版時，系統會 fail closed，而不是送出不完整或錯誤回覆。
