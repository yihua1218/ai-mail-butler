## AI Mail Butler：Gmail 整合策略與合規指南

### 概述
為 Gmail 使用者實作自動化郵件回覆，需要處理各種授權方式以及 Google 安全審查要求。本文件概述從開發到生產環境的技術路線圖。

---

### 第一階段：現行實作（SMTP 搭配應用程式密碼）
現有架構使用 **SMTP 協定**搭配 **Google 應用程式密碼**。

* **機制**：使用者在 Google 帳戶設定中產生一組 16 位數的應用程式密碼，並提供給助理使用。
* **優點**：
    * 零費用；無需安全審查。
    * 可完全掌控 Rust 原生的 SMTP 客戶端。
* **缺點**：
    * 需要使用者手動設定（必須先啟用兩步驟驗證）。
    * `From` 標頭限制為帳戶擁有者的電子郵件地址或預先驗證的別名。
* **狀態**：**適用於自架或私人使用的生產環境。**

---

### 第二階段：中期擴展（OAuth 2.0 測試模式）
為改善使用者體驗，助理可在不立即進行正式審查的情況下，過渡至 **OAuth 2.0**。

* **機制**：在 Google Cloud Console 中將應用程式保持於**「測試中」**狀態。
* **主要限制**：
    * **使用者數量**：限制最多 **100 位特定測試使用者**（必須由開發者手動加入白名單）。
    * **權杖到期**：未驗證的應用程式的更新權杖可能每 **7 天**到期，需定期重新驗證。
* **整合範疇**：
    * `https://www.googleapis.com/auth/gmail.send`（代表使用者發送郵件）。
    * `https://www.googleapis.com/auth/gmail.readonly`（讀取收到的郵件以進行處理）。
* **狀態**：**建議用於 Beta 測試及小規模試用群組。**

---

### 第三階段：生產環境與公開發布（CASA 合規）
若要大規模公開發布，應用程式必須通過 **雲端應用程式安全評估（CASA）**。

#### 受限範疇與審查
任何存取、彙總或傳輸 Gmail 資料的應用程式，均受 **受限範疇（Restricted Scope）** 政策規範。

| 審查層級 | 目標對象 | 預估年費 | 要求 |
| :------- | :------- | :------- | :--- |
| **第二級** | 公開 SaaS／高流量 | 15,000 - 75,000 美元 | 第三方滲透測試與弱點評估。 |

#### 減少合規摩擦的策略
1.  **最小化範疇**：若助理僅需回覆郵件，只申請 `gmail.send` 權限。在特定情況下，此範疇可能被歸類為「敏感」而非「受限」，從而降低審查複雜度。
2.  **使用者自備憑證**：可提供文件指引進階使用者自行建立 **Google Cloud 用戶端 ID**，將合規責任轉移至終端使用者，同時讓軟體維持開源且無需集中式審查費用。

---

### 技術實作比較

| 功能 | 應用程式密碼（SMTP） | OAuth 2.0（Gmail API） |
| :--- | :------------------- | :--------------------- |
| **安全性** | 中（永久性憑證） | 高（短期權杖） |
| **設定難易度** | 手動／對使用者較複雜 | 一鍵授權 |
| **需要審查** | 否 | 是（受限範疇須通過 CASA） |
| **API 功能** | 僅能發送／接收 | 完整的對話串、標籤、草稿功能 |

---

### 建議行動方案
1.  **保留**現有 SMTP／應用程式密碼邏輯，作為自架使用者的預設備援方案。
2.  **實作** Rust 後端的 OAuth 2.0 流程，使用 `openidconnect` 或 `oauth2` crate。
3.  **提供**設定切換選項，允許使用者輸入自己的 `CLIENT_ID` 與 `CLIENT_SECRET`，以在不依賴集中式審查應用程式的情況下享有 OAuth 的優勢。
